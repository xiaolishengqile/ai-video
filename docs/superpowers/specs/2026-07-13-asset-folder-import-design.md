# 资产文件夹批量导入设计

## 目标

让用户在项目资产库中选择人物、场景或道具类型并选择一个本地文件夹。系统将其中的图片批量导入为可直接被 AI 分镜引用的资产与子资产，并明确报告每个失败文件。

## 已确认的导入规则

1. 资产类型完全由用户在导入界面选择，系统不从文件夹名或图片内容推断类型。
2. 文件名先去扩展名，再去开头编号前缀，例如 `A-15 `、`B-01 `、`C-03 `。
3. 一张无可识别变体后缀的图片创建一个独立主资产，并创建 `initial` 子资产；该图片同时成为封面与首个可供分镜绑定的 AssetItem。
4. 仅当文件名为“主体名 + 已知变体后缀”时才归入同一主资产。首期后缀为：`三视图`、`表情图`、`正面`、`侧面`、`背面`、`多机位`、`六面展开图`、`内景`、`外景`、`夜景`、`细节`、`战斗形态`、`战斗服`。
5. 示例：
   - `A-15 地表实训发布厅.png` 创建场景资产 `地表实训发布厅` 及 `initial` 子资产。
   - `A-15 地表实训发布厅多机位.png` 归入资产 `地表实训发布厅`，创建子资产 `多机位`。
   - `秦炽川.png` 创建人物资产 `秦炽川`。
   - `秦炽川 三视图.png` 归入人物资产 `秦炽川`，创建子资产 `三视图`。
   - `秦炽川便携核心诊断仪.png` 不因包含人物名而归并，创建独立资产 `秦炽川便携核心诊断仪`。
6. 无法匹配父资产、后缀不在规则内、或规则有歧义时，按独立资产导入，绝不模糊归并。

## 交互与预览

资产库增加“导入文件夹”入口。用户先选择资产类型，再选择文件夹；前端使用目录选择能力取得图片列表，并在提交前展示预览：原文件名、归一化后的名称、导入为主资产或子资产、目标父资产及原因。用户可移除不想导入的文件，但首期不提供手动重组资产关系，避免扩大范围。

浏览器不支持目录选择时，界面退化为多图片选择；导入规则保持一致。

## 后端批量导入

新增一个 multipart 批量导入端点，接收 `projectId`、`type` 与图片列表。后端负责项目访问校验、图片格式与单文件大小校验、存储图片、根据规则创建或复用主资产、创建 AssetItem，并返回逐文件结果。

每个文件独立处理：一个文件失败不会回滚或阻止其他文件。结果包含：原文件名、相对路径（前端可提供时）、状态 `success` / `skipped` / `failed`、创建的资产和子资产 ID（成功时）、以及失败/跳过原因。

同项目、同类型、同名的主资产处理规则：

- 主资产图片遇到已存在同名资产：跳过，并报告“同名资产已存在”。
- 识别出的变体图片遇到已存在的父资产：添加为该父资产的子资产。
- 同一父资产已有同名子资产：跳过，并报告“同名子资产已存在”。

首期不做图片哈希去重、ZIP 导入、OCR/视觉识别或自动修改已有资产。

## 与 AI 分镜的关系

主资产负责项目内的名称、类型与组织；每个导入图片必须创建 AssetItem。主图创建 `initial` 子资产，因此剧本实体解析和镜头默认继承能直接取得可绑定的 AssetItem。已识别的多机位、角度和表情图是同一主资产的额外子资产，可供后续镜头显式选择。

## 验收与测试

- 名称归一化与主资产/子资产归类覆盖编号、空格、中文名称、已知后缀及未知后缀。
- 批量导入覆盖：主图创建、变体归并、未知名称独立创建、同名跳过、单文件上传失败后继续处理、无权限拒绝。
- 前端覆盖目录文件收集、预览显示、失败文件名称与原因展示；至少执行 TypeScript 类型检查。
- 复用现有图片格式限制与单文件 100MB 限制；每个失败项必须可见。
