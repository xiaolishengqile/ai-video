# 分镜流式任务稳定性设计

日期：2026-07-14

## 目标

消除长时间、多子 Agent 的分镜任务中，后端实际完成但前端任务面板显示 `network error` 的问题。

## 根因

模型推理块按 token 原样进入 SSE、Redis Stream 和重连回放。并行子 Agent 交错输出时，现有单一累积器频繁刷新，导致一次重连需要传输数万条事件；浏览器读取、JSON 解析和状态更新无法及时跟上，连接因此失败。任务本体仍可在后端完成。

## 方案

1. 不再向前端输出逐 token 的 `REASONING` 内容；保留正文、工具调用、工具完成和任务终态。
2. Replay List 最多保留 2,000 条最近逻辑事件。Redis Stream 本已只保留最近 200 条实时事件，二者的容量边界一致且可预期。
3. 初始 SSE 连接失败时，前端先查询服务端状态：
   - `COMPLETED`：直接完成任务；
   - `ACTIVE`：重连；
   - `ERROR` / `NONE`：显示错误。
4. 重连失败时重复一次状态确认，避免终态已写入 Redis 却被网络错误覆盖。

## 范围与非目标

- 不改变分镜 Agent 的并发数、工具调用、持久化逻辑或生成结果。
- 不新增数据库表或迁移。
- 不在本期展示逐 token 推理文本；推理耗时元数据仍可由现有事件路径提供。
- 不尝试回放被容量上限裁剪的旧文本；完整业务结果仍由数据库保存。

## 验收

- Redis Replay List 每次追加后被裁剪到 2,000 条以内。
- `ThinkingBlock` 不产生 SSE `REASONING` 事件。
- 初始或重连 SSE 失败后，服务端状态为 `COMPLETED` 时，前端任务状态为完成而非错误。
- 后端聚焦测试和前端 lint/build 通过。
