# AgentScope 空工具调用防御设计

日期：2026-07-15

## 目标

在不修改模型、流式输出或并行工具配置的前提下，阻止上游 OpenAI-compatible 响应中的幽灵空工具调用进入执行与前端事件流，同时保留合法无参数工具。

## 已确认根因

`script_full_parse` 任务的有效上下文包含 `scriptId`，业务工具也能正常执行；但同一轮模型响应会额外产生带独立调用 ID、参数为 `{}` 的工具调用。AgentScope 当前把这些调用与有效调用一起执行，因而产生缺少 `scriptId` 等必填参数的错误卡片。

## 方案

1. 新增一个主、子 Agent 共用的 `PostReasoningEvent` Hook，在 AgentScope 已完成流式片段聚合、尚未进入 `PreActingEvent` 前检查工具调用。
2. 仅过滤同时满足以下条件的调用：
   - 参数对象为空；
   - 对应工具 JSON Schema 的 `required` 列表非空。
3. 同一轮同时存在有效调用时，只移除无效空调用，让有效调用照常并行执行。
4. 同一轮只有无效空调用时，不把它交给执行器；向 Agent 追加一条简短纠错消息并重新推理，要求重新生成完整必填参数。
5. Schema 未声明必填参数的合法无参数工具不受影响。
6. 修正 `save_script_episode` 的 Schema，使 `scriptId`、`episodeNumber`、`title` 与执行器校验保持一致。
7. 将 `script_full_parse` 的无效 `{scriptContent}` 指令模板替换为明确的任务上下文，并在系统提示中要求所有写操作携带当前 `scriptId`。

## 数据流

模型流式响应 → AgentScope 聚合工具调用 → 空调用防御 Hook → 有效调用进入 `PreActingEvent`、工具执行和前端时间线；无效空调用在此之前被移除并触发重试。

## 错误处理与日志

- 每次过滤记录工具名、调用 ID 和 Agent 名称，不记录剧本正文或其他大参数。
- 过滤行为不伪装成工具成功，也不写入业务数据库。
- 纠错重试继续受 Agent 现有最大迭代次数和取消令牌控制。

## 测试

- 必填工具的 `{}` 调用被过滤。
- 有效调用与幽灵空调用并存时，仅保留有效调用。
- 只有幽灵空调用时，请求 Agent 重新推理。
- Schema 无必填字段的合法 `{}` 调用被保留。
- `save_script_episode` Schema 声明三个实际必填字段。
- `script_full_parse` 指令模板能够注入 `projectId` 和 `scriptId`，不再残留 `{scriptContent}`。

## 非目标

- 不关闭流式输出或并行工具调用。
- 不更换 GLM 模型、API 网关或数据库配置。
- 不修改普通业务工具的执行语义。

## 验收

- 上述聚焦测试全部通过。
- 后端完整测试通过。
- 复跑剧本解析时，幽灵 `{}` 调用不再出现在工具执行日志和前端红色卡片中，有效分集保存仍可并行执行。
