# 全局资产选择与分页稳定性修复设计

## 目标

修复全局素材资产页的三个关联问题：

- “全选当前筛选结果”覆盖全部匹配资产，而不只覆盖已经滚动加载的资产。
- 无限分页不再产生重复资产 ID 和 React 重复 key 警告。
- 批量删除后重新建立正确的分页与网格状态，剩余卡片保持正常尺寸。

## 根因

后端分页只按 `update_time DESC` 排序。数据库中大量批量生成资产拥有相同更新时间，例如资产 `1397` 与另外 109 条资产的更新时间完全相同，因此跨页边界没有唯一顺序，同一资产可能在不同请求中重复出现。

前端 `loadMore` 仅用异步 React state 作为加载锁，短时间内可能重复请求同一页，并且追加响应时不按 ID 去重。批量删除成功后，页面直接修改当前数组和总数，却保留旧的 `currentPage`；删除会改变数据库分页偏移，后续加载因此可能跳过或残留记录。重复 key 又使 React 的列表重排进入不受支持的状态，造成截图中的异常卡片布局。

## 方案

### 稳定分页

资产查询统一使用 `update_time DESC, id DESC`。更新时间保持主要排序规则，ID 只负责为相同时间的资产提供稳定次序。

前端增加同步加载锁，避免 React state 尚未提交时重复进入 `loadMore`。分页结果追加到列表时按资产 ID 去重，作为接口边界的防御。

### 全选当前筛选结果

保留普通浏览时的无限分页。只有用户点击“全选当前筛选结果”时，才复用现有 `/api/asset/all` 接口，以当前项目、类型和关键词筛选条件请求第一页，并把 `size` 设置为当前匹配总数。

请求成功后：

- 用返回的完整、去重资产列表替换当前列表；
- 将所有返回资产 ID 写入选择集合；
- 页面显示的删除数量与实际唯一 ID 数量一致；
- 再次点击全选时清空选择。

此方案不新增接口。当前数据量为数百条，只在用户明确全选时加载全部记录；如果未来单次筛选达到数万条，再升级为仅返回 ID 的专用接口。

### 删除后刷新

批量删除成功后退出选择模式、清空选择，并重新执行第一页加载。重新加载会同时重置资产数组、总数、类型统计和页码，避免在已经发生偏移的旧分页上继续加载。

网格继续使用响应式固定列数，同时为卡片增加宽度约束，确保极少记录时不会铺满内容区。

## 错误处理

- 全选请求期间禁用全选和删除按钮并显示加载状态，失败时保留现有列表和选择并显示错误提示。
- 分页加载失败保持当前列表，不推进页码，释放同步加载锁后允许重试。
- 删除失败保持选择模式和选择集合，用户可以重试。

## 测试

- 后端测试断言可访问资产分页 SQL 同时包含更新时间和 ID 的降序排序。
- 前端纯函数测试覆盖分页追加去重，以及全选数据按 ID 去重。
- 页面接线测试覆盖“全选当前筛选结果”、删除后调用第一页刷新，以及同步分页锁。
- 运行前端 Node 测试、ESLint/TypeScript 构建检查和相关后端测试。

## 非目标

- 不新增资产 ID 专用接口。
- 不改项目内资产页的非分页选择逻辑。
- 不重构全局资产页之外的资产管理功能。
